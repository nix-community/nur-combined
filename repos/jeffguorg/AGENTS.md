# AGENTS.md

## AI 协作要求

- 开始修改前，先阅读相关文件和命令输出，明确观察结果、推理过程和实施计划。
- 只修改与用户请求直接相关的文件和逻辑，不做顺手重构、格式化或无关清理。
- 不覆盖用户已有改动；遇到未预期的变更时，先判断是否影响当前任务，必要时向用户确认。
- 不使用占位代码替代真实实现，例如 `TODO`、省略号或“保留原逻辑”等不完整片段。
- 保持变更小而清晰，优先沿用项目已有的 Nix 表达式风格、命名方式和目录边界。

## Nix 变更准则

- 新增或调整包时，优先使用现有的 `callPackage` 模式，并把包实现放在合适的 `pkgs/` 路径下。
- 不在包表达式中引入不必要的全局依赖、隐式导入或环境假设。
- 需要更新源码、哈希或生成文件时，说明来源和验证方式，避免手写无法追溯的生成结果。
- 修改模块、overlay、lib 或包导出时，确认引用路径和公开属性名保持一致。

## 验证要求

- 能运行验证时，优先运行最小必要的 Nix 检查或构建命令。
- 如果无法运行验证，明确说明原因，并指出仍需用户确认的风险。
- 报告结果时只总结与任务相关的输出，不贴无关日志。

## 提交要求

- 提交前检查是否存在未 staged 变更；如果存在，停下来寻求用户或协作者确认后再继续。
- 使用 Conventional Commits 格式编写提交信息，例如 `feat: add package`、`fix: update source hash`、`docs: update agent guidance`。

## 推送后要求

- 每次 `git push` 成功后，运行 `curl -v -XPOST 'https://nur-update.nix-community.org/update?repo=jeffguorg'` 触发 NUR 更新。

## 沟通要求

- 面向用户使用简体中文。
- 先说明观察，再说明推理和计划，最后执行修改。
- 最终回复应简洁说明完成内容、验证结果和任何剩余风险。
